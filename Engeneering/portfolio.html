1. Programming Objectives

Our programming this season pursued three objectives: scoring 9 artifacts during the 30-second autonomous period using concurrent path following and intake operations, scoring them in the correct color pattern by reading AprilTag IDs 21, 22, and 23 with a Limelight 3A camera, and eliminating manual shooter adjustment in TeleOp through interpolation-based distance calculation and automatic goal alignment. Every architectural decision traces back to one of these goals.

2. Custom Kotlin DSL for Path Planning

PedroPathing provides reliable path following, but its Java API is verbose and error-prone under rapid iteration. We wrote a Kotlin DSL wrapper—housed in a separate Gradle module (PedroPathingDSL/)—that exposes the same functionality through type-safe builders, operator overloading for waypoints, and a callback system supporting temporal, parametric, and position-based triggers. Paths that previously took over ten minutes to define and debug now take under two. Compile-time checking catches configuration mistakes before they reach the robot, and the builder syntax makes each route self-documenting for the rest of the team. We did not modify PedroPathing itself; the DSL is purely a translation layer.

3. Advanced Software Architecture

Early in the season we used Java state machines, where each new behavior added another enum value and another branch in a growing switch block. Switching to Kotlin coroutines let us write sequential-looking code that runs without blocking: a CoroutineOpMode base class wraps a SupervisorJob scope so all child tasks cancel automatically when the OpMode ends, and a single subsystem failure cannot crash the others. On top of coroutines we adopted Metro for compile-time dependency injection—OpModes declare the interfaces they need, Metro resolves implementations at build time, and no subsystem knows about any other subsystem directly. State changes flow through Kotlin StateFlow and SharedFlow, giving every consumer the latest reading without polling and with built-in backpressure handling.

4. System Architecture Overview

The codebase is organized into three layers. OpModes contain zero hardware initialization; they only orchestrate subsystems injected via Metro. Subsystem interfaces define contracts (Shooter, Drive, Intake, Sorter, Vision), with implementations hidden behind those interfaces so we can swap, mock, or optimize them independently. The hardware abstraction layer talks to FTC SDK motors, servos, and sensors. This separation lets different team members develop subsystems in parallel and lets us test logic without a physical robot.

5. Shooter System: Precision Control

Scoring from variable distances (0.97–3.0 m) requires the shooter to adjust both flywheel velocity (1 540–2 170 RPM) and hood angle (0–0.33 servo position) continuously. We calibrated 10 field positions empirically and use monotone cubic spline interpolation to map any distance to smooth setpoints—avoiding the abrupt jumps that linear interpolation would cause. A combined PID + feedforward controller tracks those setpoints: the feedforward term (kS·sign(v) + kV·v) predicts the required motor power so the flywheel converges in roughly 200 ms instead of the 800 ms PID alone needed, while the PID term handles residual error from battery changes and mechanical wear. Limelight auto-alignment calculates the angle from robot pose to goal pose and sets the turret automatically, so the driver presses one button and the system handles the rest.

6. Reactive Sensor Processing

Rather than polling every loop cycle, our intake’s REV ColorRangeSensor v3 emits readings into a Kotlin Flow that debounces for 30 ms, classifies artifacts by RGB ratio thresholds (green/red > 1.5 → GREEN, red/green > 1.5 → PURPLE), filters with distinctUntilChanged, and publishes the result as a StateFlow. Multiple consumers—sorter, auto-cutoff logic, telemetry—observe the same stream without duplicating work. This pipeline reduced sensor-related CPU usage by roughly 35% and eliminated the coordination bugs we had when multiple subsystems polled the same sensor independently.

7. Design Process & Iterations

Hardware and software evolved together across three major cycles. The shooter started as a single motor running at maximum power; it physically could not reach far distances, so the mechanism was redesigned twice—improving geometry, compression, and eventually moving to a dual-motor configuration—before advanced control made sense. Only after the third hardware revision did PID + feedforward and interpolation LUTs deliver their full benefit. Localization followed a similar arc: IMU-only tracking drifted too much for repeatable autonomous, so we adopted PedroPathing with a GoBilda Pinpoint odometry computer for sub-inch accuracy. The sorter progressed through three implementation files (SorterImpl → SorterOptimized → SorterWrapped), each adding more automation—from fully manual slot changes to intelligent artifact routing with 90 ms cycle times. The color sensor was relocated from the intake ramp to directly above the sorter intake slot, improving detection reliability at the exact transfer point.

Planned next steps: dynamic path re-planning when vision detects positioning conflicts, segment-level timeout diagnostics for autonomous debugging, and predictive aiming compensation for shots taken while the robot is still moving.

The recurring lesson across all iterations: consistency outweighs peak performance. A robot that reliably scores 7 artifacts is more valuable than one that occasionally hits 9 but fails unpredictably. And every concurrent operation needs explicit timeouts and cancellation—leaving any coroutine unchecked is a match-ending risk.

8. Robot Subsystems Implementation

Drive

Mecanum drivetrain with four motors and GoBilda Pinpoint odometry (two-wheel + IMU fusion). The Drive subsystem initializes the starting pose, tracks live position throughout the OpMode, and transfers lastPose to the next OpMode so TeleOp begins exactly where autonomous ended. For autonomous, it mirrors all blue-side poses programmatically for red alliance execution—one logic source, zero duplication. Path execution chains preload delivery, collection sweeps, scoring approaches, and exit maneuvers using our DSL, with power constraints tuned per segment. During collection cycles, Drive coordinates with intake and sorter through coroutine jobs: path following and artifact pickup run in parallel, and whichever finishes first cancels the other. In TeleOp, the driver switches between field-centric and robot-centric modes, while the follower pose feeds into Limelight alignment for precision shooting.

Shooter

Dual-motor flywheel with PID + feedforward velocity control, interpolation LUT for distance-to-setpoint mapping, and Limelight-driven turret alignment. Exposes a StateFlow so other subsystems know when the flywheel has reached target velocity (within ±40 RPM). The shootAll() routine reads the sorter’s artifact order and fires in the sequence determined by the detected AprilTag pattern.

Intake + Sorter

Motor-driven roller with a REV ColorRangeSensor v3 positioned above the sorter intake slot. The reactive flow pipeline classifies GREEN vs PURPLE artifacts in real time with 30 ms debouncing and distance thresholding (<8 cm). The sorter is a 3-slot rotating carousel that tracks artifact type per slot, automatically selects the next open position, and coordinates handoff timing with both the intake and the shooter. Cycle time was optimized from 150 ms down to 90 ms through repeated field testing.

Vision

Limelight 3A camera with multiple pipeline configurations. In autonomous, it detects AprilTag IDs 21, 22, and 23 to determine the scoring pattern within a bounded detection window, falling back to a timed default if no tag is read. In TeleOp, it provides real-time targeting corrections for shooter auto-alignment.

9. Autonomous & TeleOp Programming

Both autonomous routines use alliance mirroring: a single class parameterized by Alliance.BLUE or Alliance.RED generates the correct poses automatically, halving bug surface area and ensuring changes apply to both sides.

Routine Flow
FarAuto Preload delivery → first artifact collection (concurrent intake + path) → corner artifact sweep → scoring cycles
CloseAuto Preload score → collect/shoot cycle 1 → collect/shoot cycle 2, with AprilTag pattern mapping and timed fallback

CloseAuto includes a dedicated free-gate opening path that we enable when our alliance partner’s autonomous benefits from the gate being opened, and skip when their strategy requires a cleaner cycle path.

The core concurrent pattern is followAndIntake(): two coroutine jobs—one following the collection path, one streaming detected artifacts into the sorter—race against each other and a 5-second hard timeout. Whichever completes first cancels the other, keeping every cycle bounded and deterministic.

TeleOp runs a dual-driver configuration in FullTeleOp.kt. Driver 1 handles movement (strafing, rotation, field-centric toggle, Limelight alignment). Driver 2 handles scoring (auto-shoot sequence, manual spin-up, turret/angle fine adjustment, color-specific intake, sorter positioning). The system provides haptic feedback when the shooter reaches target velocity and automatically stops the intake when the sorter is full.

10. Development & Testing Infrastructure

We wrote 13 dedicated tuning OpModes (over 1 200 lines in Tuning.kt) alongside FTC Dashboard integration for live parameter adjustment. The workflow is: deploy, open Dashboard in a browser, adjust PID gains or path parameters in real time, observe effects, record successful values, and commit. @Config annotations expose key parameters so they can be changed without redeploying code. Path visualization overlays show planned paths versus actual robot trajectory, making localization drift and path-following errors immediately visible.

Our testing philosophy centers on consistency before speed. Shooter accuracy was validated with 20+ shots from each of the 10 calibration points. Intake classification was tested across 100+ pickup attempts under varied lighting (bright, dim, partial shadow). Autonomous routines were run 10+ times each, with every failure mode documented. Only after consistency benchmarks were met did we optimize timing—sorter delays were reduced from 150 ms to 90 ms, and path power constraints were tightened—always re-validated after each change.
