2. Custom Kotlin DSL for Path Planning

TEAM INNOVATION

Our Approach

Team Xeo developed a custom Domain-Specific Language (DSL) for the PedroPathing library to address our specific workflow needs and make autonomous programming more intuitive for our team.

2.1 Why We Built This

While PedroPathing is an excellent path-following library, we found its Java-style API verbose for our development workflow. We created a Kotlin DSL wrapper to make our autonomous programming more expressive and reduce errors during development.

Standard PedroPathing API
PathChain pathChain =
    follower.pathBuilder()
    .addBezierLine(startPose, targetPose)
    .setLinearInterp(startHeading,endHeading)
    .setPathEndTimeoutConstraint(500)
    .addTemporalCallback(200) {
        intake.isRunning = true
    }
    .addParametricCallback(0.5) {
        shooter.setVelocity(2000)
    }
    .build();
Our DSL Wrapper
pathChain(constraints.copy(timeout = 500)) {
    /* Automatically adds linear
       heading interpolation
       between start and target poses
    */
    pathLinearHeading {
        +startPose
        +targetPose
        callbacks {
            temporalCallback(200.ms) {
                intake.isRunning = true
            }
            parametricCallback(0.5) {
                shooter.setVelocity(2000)
            }
        }
    }
}

2.2 Technical Implementation

Our DSL leverages Kotlin's advanced language features to create a declarative, type-safe API:

Callback System Temporal, parametric, and position-based callbacks Coordinate subsystems during path execution
Multiple Heading Modes Linear, tangent, constant, facing point interpolation Flexible robot orientation control
Operator Overloading Natural syntax using + operator for waypoints Intuitive, readable code
Builder Pattern Clean nesting structure mirrors logical flow Self-documenting code

Architecture Location: PedroPathingDSL/ - Separate Gradle module with independent build configuration, making it reusable across seasons. We plan to publish this library publicly for the FTC community to use.

  • PathChainBuilder: Top-level DSL entry point
  • PathBuilder Variants: LinearHeading, Bezier, BezierLine implementations
  • CallbackBuilder: Type-safe callback registration system
  • Extension Functions: Kotlin extensions for seamless PedroPathing integration

3.2 Coroutine-Based OpMode Architecture

KEY DECISION

The Challenge: Our initial state machine approach grew complex as we added more autonomous behaviors. We researched alternative concurrency models and discovered Kotlin coroutines. Learning how coroutines work and understanding concepts like structured concurrency and suspending functions required significant study, but this investment proved essential to solving our state management problems.

Our Initial State Machine Approach
enum State { 
    DRIVE_TO_GOAL, 
    WAIT_FOR_READY, 
    SHOOT, 
    DONE 
}
State currentState = State.DRIVE_TO_GOAL;

while (opModeIsActive()) {
    switch (currentState) {
        case DRIVE_TO_GOAL:
            if (drive.atTarget()) 
                currentState = State.WAIT_FOR_READY;
            break;
        case WAIT_FOR_READY:
            if (shooter.isReady()) 
                currentState = State.SHOOT;
            break;
        // Additional states added complexity
    }
}
Our Coroutine-Based Approach
launch {
    drive.goToGoal()
    shooter.waitUntilReady()
    shooter.shoot()
}
// Sequential code that doesn't block!
// Automatic cancellation on OpMode stop
// No state tracking needed

Result: Reduced our state management code significantly while making our autonomous routines easier to understand and modify.

Custom CoroutineOpMode Base Class

abstract class CoroutineOpMode : LinearOpMode() {
    private val scope = CoroutineScope(SupervisorJob() + Dispatchers.Default)
    
    protected fun launchAsync(block: suspend CoroutineScope.() -> Unit) {
        scope.launch { block() }
    }
    
    override fun runOpMode() {
        try {
            runOpModeImpl()
        } finally {
            scope.cancel() // Automatic cleanup
        }
    }
}
  • Non-blocking Operations: Subsystems operate concurrently without blocking main loop
  • Automatic Cancellation: All coroutines cancelled gracefully when OpMode stops
  • Structured Concurrency: Parent-child relationships prevent "leaked" background tasks
  • Exception Handling: SupervisorJob prevents one failure from cascading

Real-World Example: Concurrent Operations

withTimeout(5.seconds) {
    launch { follower.followPath(ballPath) }
    launch { intake.collectUntilFull() }
}
// Waits for both to complete or timeout

3.3 Reactive Programming with Kotlin Flows

Initial Approach: We polled sensors every loop iteration and manually managed state changes
Evolved Approach: Our sensors now expose StateFlow and SharedFlow for reactive, backpressure-aware data streams

val artefactFlow: StateFlow<ArtefactType?> = colorSensor
    .readingsFlow
    .debounce(30.milliseconds)              // Filter noise
    .map { reading -> classify(reading) }   // Process
    .stateIn(scope, SharingStarted.Eagerly, null)
Benefit Description
Declarative Express "what" not "how" - more readable code
Backpressure Handling System automatically handles fast sensor updates (prevents buffer overflow)
Composability Combine multiple sensors with operators: combine(), zip()
Memory Efficient StateFlow keeps only latest value, SharedFlow has configurable buffers

Practical Application: Shooter Ready State

shooter.stateFlow
    .filter { it.velocityError < 80 }  // RPM tolerance
    .first()                            // Wait for ready state
    .also { gamepad.rumble(200) }       // Haptic feedback

3.4 Interface-Based Subsystem Design

Every major subsystem is defined by a Kotlin interface with implementation details hidden:

interface Shooter {
    val stateFlow: StateFlow<ShooterState>
    suspend fun setDistance(meters: Double)
    fun setVelocity(rpm: Double)
    fun setHoodAngle(position: Double)
}

SOLID Design Principles

OpModes depend on interfaces, not concrete classes, following the Dependency Inversion Principle. This enables:

  • Mock implementations for testing
  • Multiple implementations for optimization (SorterImpl vs SorterOptimized)
  • Iterative development without breaking OpMode code
  • Parallel development by multiple team members

4.3 Key Design Decisions

Separation of Concerns

  • OpModes contain ZERO hardware initialization code
  • Subsystems don't know about other subsystems (communication via OpMode)
  • Testing and simulation possible without physical robot

Reactive Data Pipeline

Sensors → Flows → Processing → StateFlow

  • Backpressure handling prevents overwhelming slow consumers
  • Debouncing and filtering at subsystem level
  • No polling required - event-driven architecture

Concurrent Execution

  • Drive, intake, shooter, elevator all operate on independent coroutines
  • No blocking operations in main loop, allows for reactive controls while CPU intensive operations run in the background.
  • Graceful degradation if one subsystem fails (SupervisorJob)

5. Shooter System: Precision Control

ADVANCED CONTROL

The shooter is our most sophisticated subsystem, combining multiple advanced techniques for consistent scoring.

5.1 Problem Statement

Scoring in DECODE requires launching artifacts into a goal at varying distances (1-3 meters):

  • Variable Distance: Velocity and hood angle must adjust based on position
  • Real-time Targeting: Robot moves while aligning
  • Consistent Performance: Must hit target reliably across entire field

5.2 Interpolation Lookup Tables

The Problem: Initially, our shooter calibration used discrete distance thresholds that caused it to jump between velocity values abruptly. This created inconsistent shot trajectories at boundary distances.

Research & Solution: We researched interpolation methods and discovered that monotone cubic spline interpolation would provide smooth transitions between our calibrated points. This approach maps distance → (velocity, hood angle) continuously:

Measurement Range Data Points
Distance 0.966m → 3.0m 10 field positions
Velocity 1540 → 2170 RPM Measured empirically
Hood Angle 0 → 0.33 servo position Measured empirically

Why Cubic Spline Interpolation?

Linear interpolation: Creates discontinuous acceleration (unphysical)
Cubic splines: Smooth, continuous derivatives matching projectile motion physics

5.3 PID + Feedforward Control

Control Strategy: NextFTC Control library for flywheel velocity

Parameter Value Purpose
kP 0.003 Proportional correction
kI 0.0 Integral (not needed for velocity)
kD 0.0001 Derivative damping
kS 0.05 Static friction compensation
kV 0.012 Velocity feedforward

Why Feedforward?

PID alone: Reacts to error after it occurs (~1s settling)
Feedforward: Anticipates required power based on target velocity (~400ms settling)

Control equation: power = kS × sign(velocity) + kV × velocity + PID(error)

5.4 Limelight Auto-Alignment

suspend fun alignToGoal() {
    val goalPose = Pose2d(12.0, 132.0, 0.0) // Blue alliance goal
    val robotPose = drive.pose
    
    val angleToGoal = atan2(
        goalPose.y - robotPose.y,
        goalPose.x - robotPose.x
    )
    
    turret.setAngle(angleToGoal - robotPose.heading)
    
    val distance = hypot(
        goalPose.x - robotPose.x,
        goalPose.y - robotPose.y
    )
    
    setDistance(distance) // Applies interpolation LUTs
}

6.2 Intake Color Sensor Processing Pipeline

Hardware: REV ColorRangeSensor v3
Goal: Detect artifact color (GREEN vs PURPLE) and distance

val artefactFlow: StateFlow<ArtefactType?> = flow {
    while (currentCoroutineContext().isActive) {
        val reading = colorSensor.read()
        emit(reading)
        delay(10) // 100Hz sampling
    }
}
.debounce(30.milliseconds)          // 1. Filter vibration noise
.map { reading ->                   // 2. Classify artifact
    when {
        reading.distance > 8.0 -> null              // No artifact
        reading.alpha < 350 -> null                 // Too dark
        reading.green > reading.red * 1.5 -> GREEN
        reading.red > reading.green * 1.5 -> PURPLE
        else -> null                                // Ambiguous
    }
}
.distinctUntilChanged()             // 3. Only emit on state change
.stateIn(scope, SharingStarted.Eagerly, null)

6.4 Backpressure Management

Problem: Fast sensors (1000Hz) + slow processing (100Hz) = buffer overflow

Solution: StateFlow keeps only the latest value, automatically dropping intermediate readings that consumers can't process in time.

6.5 Sensor Fusion Example

Use Case: Combine multiple sensors to determine "ready to shoot"

val readyToShootFlow: Flow<Boolean> = combine(
    shooter.velocityFlow,      // Is flywheel at speed?
    sorter.hasArtifactFlow,    // Do we have artifact?
    turret.alignedFlow         // Are we aimed correctly?
) { velocity, hasArtifact, aligned ->
    velocity > 1900 && hasArtifact && aligned
}

// OpMode observes
readyToShootFlow.collect { ready ->
    if (ready) gamepad.rumble()

7.3 Sorter Automation Evolution

Context: Like the shooter, the sorter evolved through progressive automation as we developed more sophisticated control strategies.

Hardware Iteration: An important hardware change was relocating the REV ColorRangeSensor v3 from the intake ramp to above the sorter's intake slot. This improved detection reliability by sensing artifacts at the exact moment they enter the sorting mechanism, reducing false readings from artifacts still in the intake path.

Three Implementation Files = Three Development Stages:

Stage File Approach Features

2 SorterOptimized.kt Semi-Automated State management (track artifact in each slot), automatic positioning, some manual intervention

Why Keep All Three? Each file represents a development milestone, showing the progressive automation of what started as a simple mechanism. Similar to shooter evolution: manual → driver-assisted → fully automated.

7.5 Lessons Learned

  1. Hardware-Software Iteration: Our software benefited from hardware redesigns. The shooter needed 3 physical iterations before advanced control could show its value.
  2. Coroutines Simplified Development: Switching to coroutines was our biggest code quality improvement, reducing complexity significantly.
  3. Incremental Automation: Both shooter and sorter evolved from manual → semi-auto → fully automated. Progressive improvement worked better than trying to achieve everything at once.
  4. Measure Everything: We tracked metrics (cycle time, CPU usage, etc.) to guide our decisions and verify improvements.
  5. Wrapper Pattern: Our DSL wraps PedroPathing without modifying it, which kept concerns separated and debugging simpler.
  6. Team Specialization: Different team members owned different systems (e.g., Metro DI), which accelerated parallel development.

10.4 Testing Methodology

Our testing approach focused on empirical validation through systematic field testing rather than purely theoretical or simulated performance.

Primary Criteria: Consistency

We prioritized consistency over raw performance in all subsystems. A robot that reliably scores 7 artifacts is more valuable than one that occasionally scores 9 but fails unpredictably.

Subsystem Consistency Metric Testing Method
Shooter Hit rate across field positions 20+ shots from each calibration point, measuring success rate
Intake Artifact detection accuracy 100+ pickup attempts, tracking false positives/negatives
Autonomous Successful completion rate 10+ full routine runs, tracking failures and partial completions
Path Following Endpoint accuracy Measuring final position variance across repeated runs

Secondary Criteria: Speed

Once consistency was established, we optimized for speed without sacrificing reliability. Our primary optimization focused on the sorter mechanism:

  • Sorter Timing Optimization: Progressively reduced movement delays and tightened position tolerances
  • Cycle Time Reduction: Initial conservative timings (150ms delays) were gradually reduced to 90ms as we verified reliable operation

Empirical Calibration Process

Shooter Interpolation LUT: Collected 10 distance measurements empirically by trial and error at field positions, recording successful (velocity, hood angle) combinations. This data formed the foundation of our interpolation tables.

PID Tuning: Used FTC Dashboard for live parameter adjustment while observing robot behavior. Started with conservative gains and incrementally increased until oscillation appeared, then backed off 20-30%.

Color Sensor Thresholds: Tested with both artifact colors under various lighting conditions (bright overhead, dim room, partial shadows) to find robust RGB ratio thresholds that worked reliably in competition environments.

Testing Philosophy

Every feature was validated through repeated field tests before being considered "competition-ready." We maintained a testing log tracking failure modes, environmental conditions, and configuration changes. This empirical approach ensured our software performed reliably under real-world conditions, not just in ideal scenarios.
